## TASK 2 — Square Cut G-code Fixes

After Task 1 confirms which file is canonical, edit `causticforge_v2.py` (the canonical source in `BlenderAddons/causticforge/`) to fix three issues in the square cut g-code generator. Find the function that produces the square cut output (search for "Square Cut" or "squarecut" in the file).

### Issue 1 — Eliminate retracts between Z layers

**Current behavior:** After cutting the perimeter at one Z depth, the toolpath retracts to safe Z (Z=0.5"), rapids back to the start corner, then plunges down to the next Z. This wastes time and stresses the bit.

**Desired behavior:** The bit is already at the start corner when each layer's perimeter closes. Just plunge straight down to the next Z and cut the next layer. No retract between layers. Only retract at the very end after the final layer.

**Sequence per layer:**
1. (Already at start corner from previous layer's end, or from initial positioning)
2. Plunge Z down to next layer depth at PLUNGE feed (F36, not cut feed)
3. Cut perimeter at that Z
4. End at start corner (the perimeter is closed — verify this)
5. Repeat for next layer
6. After final layer: retract to safe Z

**Prerequisite verification:** Each layer's perimeter MUST start and end at the same XY point. Check the current generator. If perimeters end somewhere other than the start corner, that needs fixing first — otherwise the no-retract optimization can't work.

### Issue 2 — Tab plunge rate

**Current behavior:** When the toolpath rises over a tab (Z+ move) and drops back down (Z- move) on the other side, BOTH moves use the cutting feed rate (F120). The Z- "punch through" back into the cut at full speed is hard on the bit and the work.

**Desired behavior:** Pick whichever option is simpler given the current code structure:

**Option A (simpler, default):** Use the plunge feed rate (F36) for any Z- move that re-enters the cut after going over a tab. The Z+ move going up over the tab can stay at cut feed.

**Option B (better but more code):** Ramp the Z- move. Instead of a vertical plunge back to cutting depth, ramp down over a short XY distance (e.g., 0.05–0.1") so the bit eases back into the material. Same on the way up — ramp out instead of vertical lift. This eliminates plunges entirely on tabs.

**Default to Option A** unless Option B is straightforward. Document which one was chosen in a code comment.

### Issue 3 — Program start sequence

**Current behavior:**
```
G54
G20
G17 G90

M03 S20000
G01 Z0.5000 F220.0
```
Spindle starts BEFORE the bit retracts. If the bit is sitting in the work (Z=0 or below), spindle-on with bit-in-material is bad.

**Desired behavior:** Retract to safe Z FIRST, then start spindle:
```
G54
G20
G17 G90

G01 Z0.5000 F220.0   (retract to safe height first)
M03 S20000           (then spindle on)
```

If NK105 supports a dwell after spindle-on (G04 P2 or similar) to let the spindle reach speed, add it. If you're not sure, omit the dwell — Bland can handle it manually.
